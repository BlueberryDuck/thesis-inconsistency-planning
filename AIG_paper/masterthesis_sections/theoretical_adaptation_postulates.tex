\subsection{Rationality Postulates}
\label{sec:postulates}

A core component of inconsistency measurement research is analyzing which rationality postulates measures satisfy~\cite{Hunter2008,Thimm2019}. Following~\cite{Ulbricht2016}, we recognize that domain-specific adaptations may require modified postulates. Classical postulates cannot be applied directly to planning problems for two reasons: first, planning has fundamentally different structure than propositional knowledge bases (goals, operators, states versus formulas); second, planning exhibits inherently non-monotonic behavior (adding operators can reduce inconsistency by enabling new solution paths).

\subsubsection{Planning-Adapted Postulates}

\begin{defn}[Planning Consistency]
    An inconsistency measure $I$ for planning problems satisfies \emph{Planning Consistency} if:
    \begin{equation*}
        I(\langle P, O, I, G \rangle) = 0 \text{ iff } \langle P, O, I, G \rangle \text{ is solvable}
    \end{equation*}
\end{defn}

This adapts the classical Consistency postulate ($I(K) = 0$ iff $K$ is consistent) to the planning domain, where solvability replaces logical consistency.

\begin{defn}[Operator Monotonicity]
    An inconsistency measure $I$ satisfies \emph{Operator Monotonicity} if for all planning problems $\langle P, O, I, G \rangle$ and operators $o$:
    \begin{equation*}
        I(\langle P, O \cup \{o\}, I, G \rangle) \leq I(\langle P, O, I, G \rangle)
    \end{equation*}
\end{defn}

Note that Operator Monotonicity reverses classical monotonicity because adding operators can only enable new paths, potentially reducing inconsistency. This parallels~\cite{Ulbricht2016}'s observation that non-monotonic logics require inverted monotonicity postulates.

\begin{defn}[Goal Monotonicity]
    An inconsistency measure $I$ satisfies \emph{Goal Monotonicity} if for all planning problems $\langle P, O, I, G \rangle$ and goals $g$:
    \begin{equation*}
        I(\langle P, O, I, G \cup \{g\} \rangle) \geq I(\langle P, O, I, G \rangle)
    \end{equation*}
\end{defn}

Goal Monotonicity preserves the direction of classical monotonicity: adding requirements (goals) can only increase or maintain inconsistency, never decrease it.

\subsubsection{Postulate Analysis: Unreachability Profile}

\begin{prop}
    $I^{\text{scope}}_{\text{UR}}$ does not satisfy Planning Consistency.
\end{prop}
\begin{proof}
    Counter-example: The Light Switch problem has $I^{\text{scope}}_{\text{UR}} = 0$ (both goals \textit{on} and \textit{off} appear in reachable states) but is unsolvable (goals are mutex). Conversely, Planning Consistency requires $I = 0$ only for solvable problems.
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{UR}}$ satisfies Operator Monotonicity.
\end{prop}
\begin{proof}
    Adding an operator $o$ can only expand $S_{\text{reach}}$ or leave it unchanged. If goal $g$ was unreachable (not appearing in any state in $S_{\text{reach}}$), adding $o$ may make it reachable by enabling new state transitions. This can only reduce or maintain $I^{\text{scope}}_{\text{UR}}$, never increase it.
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{UR}}$ satisfies Goal Monotonicity.
\end{prop}
\begin{proof}
    Adding goal $g$ to $G$ results in exactly one of two cases: (a) $g$ appears in some state in $S_{\text{reach}}$, so $I^{\text{scope}}_{\text{UR}}$ is unchanged, or (b) $g$ does not appear in any state in $S_{\text{reach}}$, so $I^{\text{scope}}_{\text{UR}}$ increases by exactly 1.
\end{proof}

\subsubsection{Postulate Analysis: Goal Mutex Profile}

\begin{prop}
    $I^{\text{scope}}_{\text{MX}}$ does not satisfy Planning Consistency.
\end{prop}
\begin{proof}
    Counter-example: The Locked Door problem has $I^{\text{scope}}_{\text{MX}} = 0$ (no mutex relationships exist since the goal \textit{in\_room} is not even achievable) but is unsolvable.
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{MX}}$ does not satisfy Operator Monotonicity in general.
\end{prop}
\begin{proof}
    Adding an operator can introduce new reachable states, potentially creating new mutex relationships. Consider a problem where goal $g_2$ is initially unreachable (hence no mutex with $g_1$). Adding an operator that makes $g_2$ reachable but in states that never contain $g_1$ creates a new mutex pair, increasing $I^{\text{scope}}_{\text{MX}}$.
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{MX}}$ satisfies Goal Monotonicity.
\end{prop}
\begin{proof}
    Adding goal $g$ to $G$ can only increase the set of potential mutex pairs. If $g$ is mutex with existing goals, $I^{\text{scope}}_{\text{MX}}$ increases; otherwise it remains unchanged.
\end{proof}

\subsubsection{Postulate Analysis: Goal Sequencing Conflict Profile}

\begin{prop}
    $I^{\text{scope}}_{\text{GS}}$ does not satisfy Planning Consistency.
\end{prop}
\begin{proof}
    Counter-example: The Light Switch problem has $I^{\text{scope}}_{\text{GS}} = 0$ (no sequencing conflicts since actions are reversible) but is unsolvable (goals are mutex).
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{GS}}$ satisfies Operator Monotonicity.
\end{prop}
\begin{proof}
    Adding operator $o$ can only provide new paths from $g_1$-satisfying states. If $(g_1, g_2)$ was a sequencing conflict (no path from any $g_1$-state reaches $g_2$), adding $o$ may create such a path, eliminating the conflict. Adding operators cannot create new sequencing conflicts since it can only expand reachability from each state.
\end{proof}

\begin{prop}
    $I^{\text{scope}}_{\text{GS}}$ satisfies Goal Monotonicity.
\end{prop}
\begin{proof}
    Adding goal $g$ to $G$ can only increase the set of potential sequencing conflicts. New conflicts of the form $(g, g')$ or $(g', g)$ may emerge; existing conflicts are unaffected.
\end{proof}

\subsubsection{Measure Relationships}

Beyond postulate compliance, formal propositions clarify how the three measures relate to each other, demonstrating they capture complementary aspects of planning inconsistency.

\begin{prop}[Unreachability Precludes Mutex and Sequencing]
    \label{prop:unreachable_precludes}
    Let $\langle P, O, I, G \rangle$ be a planning problem and $g \in G$.
    If $g$ is unreachable (i.e., $\nexists s \in S_{\text{reach}} : g \in s$), then:
    \begin{enumerate}
        \item $g$ does not participate in any mutex relationship
        \item $g$ does not participate in any sequencing conflict
    \end{enumerate}
\end{prop}
\begin{proof}
    Both mutex (Definition~\ref{defn:mutex}) and sequencing conflict (Definition~\ref{defn:sequencing}) require
    as their first condition that the goal proposition be achievable:
    $\exists s \in S_{\text{reach}} : g \in s$. If $g$ is unreachable,
    this condition fails, precluding both relationship types.
\end{proof}

This proposition establishes a precedence relationship: unreachability analysis should be performed first, as unreachable goals are excluded from mutex and sequencing analysis by definition.

\begin{prop}[Mutex and Sequencing are Distinct Phenomena]
    \label{prop:mutex_sequencing_distinct}
    The following hold:
    \begin{enumerate}
        \item Goals can be mutex without having a sequencing conflict
        \item Goals can have a sequencing conflict without being mutex
    \end{enumerate}
\end{prop}
\begin{proof}
    \textbf{(1)} Light Switch: Goals $\{\textit{on}, \textit{off}\}$ are mutex
    (both achievable, never co-occur) but have no sequencing conflict
    (actions are reversible; from any \textit{on}-state, \textit{off} is reachable and vice versa).

    \textbf{(2)} One-Way Door: $(\textit{has\_item}, \textit{other\_task\_done})$ is a
    sequencing conflict, but the goals are not mutex: mutex requires
    goals to never co-occur in any reachable state, while sequencing conflicts
    concern trajectory constraints (achieving one blocks the other).
\end{proof}

\begin{prop}[Scope-Structure Bounds]
    \label{prop:scope_structure_bounds}
    For any planning problem $\langle P, O, I, G \rangle$:
    \begin{enumerate}
        \item $I^{\text{scope}}_{\text{UR}} \leq I^{\text{struct}}_{\text{UR}} \leq |P|$
        \item $I^{\text{struct}}_{\text{MX}} \leq \binom{I^{\text{scope}}_{\text{MX}}}{2}$
        \item $I^{\text{struct}}_{\text{GS}} \leq I^{\text{scope}}_{\text{GS}} \times (I^{\text{scope}}_{\text{GS}} - 1)$
    \end{enumerate}
\end{prop}
\begin{proof}
    \textbf{(1)} Unreachable goals are a subset of all unreachable propositions,
    which are a subset of $P$.

    \textbf{(2)} The maximum number of unordered pairs among
    $I^{\text{scope}}_{\text{MX}}$ goals is $\binom{I^{\text{scope}}_{\text{MX}}}{2}$.

    \textbf{(3)} The maximum number of ordered pairs among
    $I^{\text{scope}}_{\text{GS}}$ goals is $n(n-1)$ where
    $n = I^{\text{scope}}_{\text{GS}}$.
\end{proof}

These bounds enable diagnostic interpretation through ratio analysis:

\begin{itemize}
    \item \textbf{Unreachability depth}: The ratio $(I^{\text{struct}}_{\text{UR}} - I^{\text{scope}}_{\text{UR}}) / I^{\text{scope}}_{\text{UR}}$ indicates cascading dependency depth. A ratio of 0 means goals are directly unreachable (no intermediate propositions blocked); larger ratios indicate deeper precondition chains.

    \item \textbf{Mutex density}: If $I^{\text{struct}}_{\text{MX}} = \binom{I^{\text{scope}}_{\text{MX}}}{2}$, the mutex goals form a \emph{complete clique} where every pair is mutually exclusive (as in Traffic Light). Ratios below 1 indicate sparser conflict structures with isolated mutex pairs.

    \item \textbf{Sequencing symmetry}: If $I^{\text{struct}}_{\text{GS}} = I^{\text{scope}}_{\text{GS}} \times (I^{\text{scope}}_{\text{GS}} - 1)$, all sequencing conflicts are \emph{bidirectional} (achieving any involved goal blocks all others, and vice versa). Ratios near 0.5 suggest predominantly unidirectional conflicts; lower ratios indicate star-shaped conflict patterns where one goal blocks many others but not conversely.
\end{itemize}

\paragraph{Diagnostic Profile Interpretation.}

The complete diagnostic profile for a planning problem is the 6-tuple:
\begin{equation*}
    \text{Profile} = (I^{\text{scope}}_{\text{UR}}, I^{\text{struct}}_{\text{UR}}, I^{\text{scope}}_{\text{MX}}, I^{\text{struct}}_{\text{MX}}, I^{\text{scope}}_{\text{GS}}, I^{\text{struct}}_{\text{GS}})
\end{equation*}

Each scenario in Table~\ref{tab:measure_values} exhibits a characteristic profile signature:
\begin{itemize}
    \item \textbf{Unreachability-dominant}: $(1, 3, 0, 0, 0, 0)$ for Locked Door indicates dependency chain failure with depth 2 (three unreachable propositions blocking one goal)
    \item \textbf{Mutex-dominant}: $(0, 0, 2, 1, 0, 0)$ for Light Switch indicates a single mutex pair involving both goals; $(0, 0, 3, 3, 0, 0)$ for Traffic Light indicates a complete mutex clique (all $\binom{3}{2} = 3$ pairs are mutex)
    \item \textbf{Sequencing-dominant}: $(0, 0, 0, 0, 2, 1)$ for One-Way Door indicates unidirectional blocking; $(0, 0, 0, 0, 2, 2)$ for Bidirectional indicates symmetric blocking (both directions)
\end{itemize}

The profile enables immediate identification of the dominant failure mode. Within each mode, the scope-to-structure ratio provides additional diagnostic depth: for mutex, $I^{\text{struct}}_{\text{MX}} / \binom{I^{\text{scope}}_{\text{MX}}}{2}$ approaching 1 indicates dense conflict structure, while for sequencing, $I^{\text{struct}}_{\text{GS}} / (I^{\text{scope}}_{\text{GS}} \times (I^{\text{scope}}_{\text{GS}} - 1))$ approaching 1 indicates symmetric (bidirectional) conflicts.

The observation that no single measure satisfies Planning Consistency motivates the multi-faceted diagnostic approach adopted in this work. As established in Propositions~\ref{prop:unreachable_precludes}--\ref{prop:scope_structure_bounds}, the measures form a complementary hierarchy: unreachability analysis takes precedence (Proposition~\ref{prop:unreachable_precludes}), mutex and sequencing capture distinct phenomena (Proposition~\ref{prop:mutex_sequencing_distinct}), and scope-structure bounds enable diagnostic interpretation (Proposition~\ref{prop:scope_structure_bounds}). This design parallels findings in~\cite{Ulbricht2016}, where multiple measures with different postulate profiles were necessary to adequately characterize inconsistency in answer set programs.
